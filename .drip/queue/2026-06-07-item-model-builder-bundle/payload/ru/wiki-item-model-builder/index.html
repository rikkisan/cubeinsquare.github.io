Как собирать item definitions для Minecraft 1.21.4+

Сложность новой папки items не в том, что файл большой. Сложность в том, чтобы каждый раз выбрать правильный тип ветвления. Этот гайд как раз про этот выбор.

Открыть конструктор item_model Открыть генератор паков

Зачем вообще появились item definitions

Minecraft 1.21.4 вынес логику отрисовки предметов в отдельные файлы assets/<namespace>/items/. То есть старое чувство “сейчас всё впихну в overrides” больше не даёт чистый результат.
Новая система стала честнее: один файл может быть прямой model, select-деревом, range_dispatch по порогам или условным condition. Плюс в ясности, минус в том, что ручной каркас быстро надоедает.

Подбирайте ветку под саму задачу

ВеткаКогда подходитТипичный пример
minecraft:modelНужна одна фиксированная модель.Пропуск на базе paper.
minecraft:selectОдно свойство выбирает между именованными ветками.Строковые состояния custom_model_data.
minecraft:range_dispatchЧисло пересекает пороги.Стадии урона или заряд.
minecraft:conditionМодель зависит от true/false.Выбранный предмет или использование предмета.

Практический workflow

  1. Сначала определите путь файла внутри items/.
  2. Выберите самый маленький тип ветвления, который реально объясняет логику.
  3. Сначала настройте fallback, потом добавляйте ветки.
  4. Скопируйте JSON и проверьте предмет в игре до того, как размножать паттерн по всему паку.

Типичные ошибки

  • Брать select там, где хватило бы обычной model.
  • Забывать fallback и потом терять базовое чтение файла.
  • Смешивать ожидания строковых и числовых значений в одной definition.
  • Называть файл и model path так по-разному, что команда уже не понимает, что к чему относится.

FAQ

Нужен ли select всегда, если речь про CustomModelData?

Нет. Если у предмета только один внешний вид, обычный model-файл проще. Select нужен, когда один definition должен ветвиться на несколько обликов.

Когда range_dispatch лучше, чем select?

Когда решение числовое и упорядоченное: урон, кулдаун, заряд или любое пороговое значение.

Почему начинать с fallback?

Потому что fallback — это базовое чтение предмета. Когда оно правильно, все остальные ветки уже легче оценивать.

Можно ли держать item-команду отдельно от model-файла?

Да, и это одна из сильных сторон новой системы. Конструктор отвечает только за визуальную definition-сторону.

Как это связано с остальным сайтом?

Item model builder собирает definition JSON, custom item builder — /give-команду, а resource-pack generator помогает упаковать визуальные файлы.
