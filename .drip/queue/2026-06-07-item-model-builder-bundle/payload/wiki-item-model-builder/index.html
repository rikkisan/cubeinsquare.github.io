How to build item definitions for Minecraft 1.21.4+

The hard part of the new items folder is not the file itself. It is remembering which branch type matches the problem in front of you. This guide is here to keep that choice small and deliberate.

Open item model builder Open pack generator

Why item definitions exist now

Minecraft 1.21.4 moved item rendering logic into dedicated files under assets/<namespace>/items/. That means the old instinct of stuffing everything into overrides is no longer the clean path.
The new system is more explicit: one file can be a direct model, a select tree, a range-based dispatch, or a condition check. The upside is clarity. The downside is that hand-writing the wrapper gets repetitive very quickly.

Pick the branch that matches the decision

BranchUse it whenTypical example
minecraft:modelYou only need one fixed model.A plain custom paper pass.
minecraft:selectOne property chooses between named branches.Different custom_model_data string states.
minecraft:range_dispatchA number crosses thresholds.Damage stages or charge levels.
minecraft:conditionThe model depends on true/false state.Selected item, using item, damaged item.

Practical workflow

  1. Decide the file path you want inside items/.
  2. Choose the smallest branch type that explains the logic.
  3. Set a fallback first, then add branches.
  4. Copy the generated JSON and test the item in game before you duplicate the pattern across ten more definitions.

Common mistakes

  • Using select when a direct model file would do.
  • Forgetting a fallback model, which leaves the file harder to reason about later.
  • Mixing string-style and numeric-style expectations inside the same definition.
  • Naming the file and the model path so differently that the team can no longer guess what belongs to what.

FAQ

Do I always need select for CustomModelData?

No. If the item only has one fixed look, a direct model file is simpler. Use select when the same definition must branch into multiple appearances.

When is range_dispatch better than select?

When the decision is numeric and ordered: damage, cooldown, charge, or threshold-like values.

Why start with the fallback?

Because the fallback is the default reading of the item. Once that anchor is correct, every extra branch becomes easier to judge.

Can I still keep item commands separate from the model file?

Yes. That separation is one of the main strengths of the new system. The builder just handles the visual definition side.

Where does this connect to the rest of the site?

Use the item model builder for the definition JSON, the custom item builder for /give commands, and the resource-pack generator when the visual files still need to be packed.
