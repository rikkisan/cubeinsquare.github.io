Comment construire des item definitions pour Minecraft 1.21.4+

Le vrai piège du dossier items n’est pas sa taille. C’est le choix de la bonne branche à chaque fois. Ce guide sert à rendre ce choix clair.

Ouvrir le constructeur item_model Ouvrir le générateur de pack

Pourquoi les item definitions existent maintenant

Minecraft 1.21.4 a déplacé la logique de rendu des objets dans des fichiers séparés sous assets/<namespace>/items/. L’ancien réflexe “tout faire dans overrides” n’est plus la voie propre.
Le nouveau système est plus explicite : un fichier peut être une model simple, un arbre select, un range_dispatch basé sur des seuils ou une condition booléenne. La clarté est meilleure, mais l’ossature répétitive fatigue vite.

Choisir la branche selon la vraie décision

BrancheQuand l’utiliserExemple
minecraft:modelUne seule apparence fixe suffit.Un pass en papier.
minecraft:selectUne propriété choisit entre plusieurs branches nommées.États string de custom_model_data.
minecraft:range_dispatchUne valeur numérique franchit des seuils.Niveaux d’usure ou de charge.
minecraft:conditionLe modèle dépend d’un vrai/faux.Objet sélectionné ou utilisé.

Workflow pratique

  1. Choisissez d’abord le chemin de fichier dans items/.
  2. Prenez la plus petite branche qui raconte la bonne logique.
  3. Réglez le fallback avant d’ajouter des branches.
  4. Copiez le JSON et testez l’objet en jeu avant de dupliquer le motif partout.

Erreurs fréquentes

  • Utiliser select là où un simple model suffit.
  • Oublier le fallback et perdre la lecture par défaut du fichier.
  • Mélanger des attentes string et numériques dans la même definition.
  • Nommer le fichier et le modèle de manière trop différente, au point que l’équipe ne sait plus ce qui correspond à quoi.

FAQ

Faut-il toujours utiliser select pour CustomModelData ?

Non. Si l’objet n’a qu’une seule apparence, un fichier model direct est plus simple. Select sert quand une même definition doit bifurquer vers plusieurs rendus.

Quand range_dispatch vaut-il mieux que select ?

Quand la décision est numérique et ordonnée : usure, charge, cooldown, seuils.

Pourquoi commencer par le fallback ?

Parce que le fallback est la lecture par défaut de l’objet. Une fois ce socle juste, chaque branche supplémentaire est plus facile à juger.

Puis-je garder la commande item séparée du fichier modèle ?

Oui. Cette séparation est justement l’un des gros avantages du nouveau système.

Comment cela se relie-t-il au reste du site ?

Le builder item_model génère le JSON de définition, le custom item builder prépare la commande /give, et le générateur de pack gère l’emballage visuel.
